Understanding IAM

What is IAM?

Identity and Access Management

IAM is a global service — it controls who can access your AWS resources and what they can do with them. Every API call in AWS goes through IAM for authorization. No exceptions.

Key Point: When you create an AWS account, you get a Root Account. This has unlimited access. You should never use it for daily tasks — create IAM users instead.

What IAM Controls

Authentication — proving who you are

Authorization — what you're allowed to do

Audit — tracking what was done

Why It Matters

Security — least privilege access

Compliance — who accessed what, when

Organization — manage teams at scale

Core Components

The Building Blocks of IAM

Users

A person or application that interacts with AWS. Each user gets unique credentials.

One physical person = One IAM user. Never share credentials.

Groups

A collection of users. Attach policies to the group, and all members inherit them.

Groups can only contain users — not other groups.

Policies

JSON documents that define permissions. They specify what actions are allowed or denied.

Apply the Least Privilege Principle — give only needed access.

Roles

Temporary permissions for AWS services or external users. No permanent credentials.

EC2 instances use roles — never store keys on instances.

Policy Deep Dive

IAM Policy Structure

IAM Policy — JSON Example
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/*"
    },
    {
      "Effect": "Deny",
      "Action": "s3:DeleteObject",
      "Resource": "*"
    }
  ]
}

Effect

Allow or Deny — what happens when the statement matches. Deny always wins over Allow.

Action

The specific API calls being controlled. Format: service:Action — like s3:GetObject or ec2:StartInstances

Resource

Which AWS resource this applies to. Uses ARN format. Use * for all resources (not recommended).

Policy Types

Different Ways to Attach Policies

AWS Managed

Pre-built by AWS for common use cases. Easy to use but less granular.

AdministratorAccess
ReadOnlyAccess
AmazonS3FullAccess

Customer Managed

Created by you. Full control over permissions. Best practice for production.

Recommended: Create your own policies tailored to your exact needs.

Inline Policies

Embedded directly in a single user, group, or role. Deleted when the entity is deleted.

Exam Tip: Inline policies have a strict 1-to-1 relationship with the entity.
Evaluation Logic: By default, all requests are denied. An explicit Allow overrides the default Deny. But an explicit Deny always overrides any Allow. This is critical for the exam.
Security Features

Protecting Your AWS Account

MFA — Multi-Factor Authentication

Password alone is not enough. MFA adds a second layer using something you physically have.

Virtual MFA — Google Authenticator, Authy
Hardware Key — YubiKey, Gemalto

Password Policy

Set rules for all IAM users in your account to enforce strong passwords.

Minimum length, require uppercase, numbers, symbols
Prevent password reuse, force rotation

Access Keys

Used for programmatic access (CLI / SDK). Each user can have max 2 active keys. Treat them like passwords — never share or commit to code.

IAM Credentials Report

Account-level report listing all users and their credential status. Use it to audit MFA, access keys, and last login times.

AWS CLI

Command Line Interface

What is AWS CLI?

A tool to interact with AWS services from your terminal. Everything you can do in the Console, you can do with the CLI — and automate it.

Three Ways to Access AWS

Console — Browser, protected by password + MFA
CLI — Terminal, protected by Access Keys
SDK — Code, protected by Access Keys
Terminal — Setup & Usage
# Configure your CLI
$ aws configure
  AWS Access Key ID: AKIA...
  Secret Access Key: wJalr...
  Default region: us-east-1

# Test it — list your S3 buckets
$ aws s3 ls

# List IAM users
$ aws iam list-users

# Describe EC2 instances
$ aws ec2 describe-instances
Never store Access Keys on an EC2 instance. Use IAM Roles instead — they provide temporary credentials automatically.
Roles Deep Dive

IAM Roles — Temporary Access

When to Use Roles?

Roles are designed for delegation — granting temporary permissions without sharing credentials.

EC2 Instance needs to read S3
Lambda Function needs to write to DynamoDB
Cross-Account access between AWS accounts
Exam Favorite: Any time you see "EC2 needs access to S3" — the answer is IAM Role, not Access Keys.

Best Practices

IAM Security Best Practices

01

Never use Root Account

Only use root for initial setup. Lock it with MFA and forget about it.

02

Enable MFA everywhere

Especially on root account and all admin users. Non-negotiable.

03

Least Privilege Principle

Give users only the permissions they absolutely need. Start with zero and add.

04

Use Groups for permissions

Never attach policies directly to users. Always use groups for team management.

05

Use Roles for services

EC2, Lambda, ECS — always use roles. Never put Access Keys on instances.

06

Audit regularly

Use IAM Credentials Report and Access Advisor to review and remove unused permissions.

Key Takeaways

Remember These for the
Exam

IAM is Global

Not region-specific. Users, groups, roles are available across all regions.

Explicit Deny Wins

If any policy says Deny, it overrides ALL Allow statements. Always.

Roles over Keys

If a question asks "EC2 needs S3 access" — answer is Role, not Access Keys.

Policy Evaluation

Default deny → Check all policies → Any explicit deny? Denied. → Any allow? Allowed. → Otherwise denied.

STS — Security Token Service

Behind the scenes, roles use STS to generate temporary credentials (AssumeRole).

Access Advisor

Shows which services a user accessed and when. Use it to refine and tighten policies.
