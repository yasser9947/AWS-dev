Understanding ELB

What is a Load Balancer?

Elastic Load Balancing

A Load Balancer is a server that sits in front of your instances and distributes incoming traffic across multiple targets. Instead of users hitting one server directly, they hit the Load Balancer, and it decides which server handles the request.

Key Point: ELB is a managed service — AWS handles the availability, upgrades, and scaling of the load balancer itself. You just configure it.

Why Use a Load Balancer?

Spread load — no single server gets overwhelmed

Single endpoint — users see one URL, not individual servers

High availability — if one server dies, traffic goes to the others

Health Checks

The ELB pings your instances regularly to check if they're alive. If an instance fails the health check, ELB stops sending traffic to it automatically.

Health check = HTTP request to a specific endpoint (e.g. /health on port 80). Response 200 = healthy.

Types of Load Balancers

Three Types of ELB

ALB

Application Load Balancer

Works at Layer 7 (HTTP/HTTPS). Understands URLs, headers, cookies. The smartest type.

Route by URL path: /api vs /images
Route by hostname: api.site.com vs web.site.com
Route by query string: ?platform=mobile
Best for: Web apps, microservices, containers

NLB

Network Load Balancer

Works at Layer 4 (TCP/UDP). Extremely fast. Handles millions of requests per second.

Ultra-low latency (~100ms vs ~400ms ALB)
Has a static IP per AZ (Elastic IP)
Preserves client source IP
Best for: Gaming, IoT, extreme performance needs

GLB

Gateway Load Balancer

Works at Layer 3 (Network). Routes traffic through third-party virtual appliances.

Firewalls, intrusion detection
Deep packet inspection
Uses GENEVE protocol on port 6081
Best for: Security appliances, network monitoring
ALB Deep Dive

Application Load Balancer — Routing

URL Path Routing

Route traffic based on the URL path. One ALB handles multiple apps.

example.com/api/*    → API servers
example.com/images/* → Image servers
example.com/*        → Default servers

Hostname Routing

Different subdomains go to different target groups.

api.example.com  → API servers
web.example.com  → Web servers
admin.example.com→ Admin panel

Important ALB Features

Fixed hostname — ALB gets a DNS name like xxx.region.elb.amazonaws.com X-Forwarded-For — Header that contains the real client IP (since ALB sits in between) X-Forwarded-Port — Original port the client used X-Forwarded-Proto — Original protocol (HTTP or HTTPS)

Exam Tip: The application server sees the ALB's private IP, not the client's IP. To get the client's real IP, read the X-Forwarded-For header.
Target Groups

Where Does Traffic Go?

A Target Group is a collection of targets that receive traffic from the Load Balancer

The ELB doesn't send traffic to individual instances — it sends to a Target Group. The Target Group then routes to healthy targets using the configured algorithm.

Target Types

EC2 Instances — most common, managed by ASG
IP Addresses — must be private IPs, for on-premise servers
Lambda Functions — ALB converts HTTP to JSON event
Another ALB — chain load balancers (NLB → ALB)

Health Check Settings

Each Target Group has its own health check configuration:

Protocol:     HTTP
Path:         /health
Port:         80
Healthy:      5 consecutive passes
Unhealthy:    2 consecutive fails
Interval:     30 seconds
Key Concept: ELB routes to Target Groups. Target Groups route to targets. One ALB can have multiple Target Groups based on different rules (path, hostname, etc.).
Auto Scaling

Auto Scaling Groups (ASG)

Scale Your Servers Automatically

ASG automatically adds or removes EC2 instances based on demand. High traffic? More servers. Low traffic? Fewer servers. You pay only for what you use.

ASG Configuration

Minimum — never go below this (e.g. 2)
Desired — what you want running normally (e.g. 3)
Maximum — never go above this (e.g. 6)
ASG works with ELB — new instances are automatically registered in the Target Group.

Launch Template

ASG needs to know how to create new instances. The Launch Template defines:

AMI — which image to use
Instance Type — t2.micro, m5.large, etc.
Security Groups, Key Pair, IAM Role
User Data — startup script
Scaling Policies

How Does ASG Decide to Scale?

Target Tracking

The simplest and most common. You set a target, ASG maintains it automatically.

# Example:
"Keep average CPU at 40%"

# CPU goes to 70%?
→ ASG adds instances

# CPU drops to 20%?
→ ASG removes instances
Most used. Like setting AC to 22 degrees — it adjusts automatically.

Step / Simple Scaling

You define rules based on CloudWatch alarms. More control but more setup.

# Step Scaling rules:
CPU > 70% → add 2 instances
CPU > 90% → add 4 instances
CPU < 30% → remove 1 instance
Step scales in steps based on alarm severity. Simple waits for cooldown between actions.

Scheduled Scaling

Scale based on known patterns. You know traffic spikes at 9 AM every day? Schedule more instances before that.

"Every weekday at 8:30 AM"
→ set min capacity to 5

"Every Friday at 6 PM"
→ set min capacity to 2

Predictive Scaling

Uses machine learning to analyze historical traffic and predict future demand. Provisions instances ahead of time.

Best for: Cyclical patterns that repeat daily or weekly. AWS learns and pre-scales.

Key Takeaways

Remember These for the
Exam

ALB = Layer 7 (HTTP)

Routes by URL, hostname, query string. Best for web apps and microservices. Gets a DNS name, not a static IP.

NLB = Layer 4 (TCP)

Ultra-fast, static IP per AZ. Use when you need extreme performance or a fixed IP address.

X-Forwarded-For

ALB replaces client IP with its own. Read X-Forwarded-For header to get the real client IP.

ASG + ELB = Best Friends

ASG automatically registers/deregisters instances with the ELB Target Group.

Target Tracking = Simplest

"Keep CPU at 40%" — ASG handles the rest. Most recommended scaling policy.

Health Check Failure

ELB marks instance unhealthy → stops traffic. ASG can terminate it and launch a replacement.
